# Stress test — running the concept-graph methodology on a *decision* domain

**What we're building:** a learning map for *"decide & set up your team's shared AI infrastructure
(Claude Code + Codex)."* **Second purpose (this file):** the domain is a **decision / best-practices**
one, not a *conceptual* one (Gödel, KG, category theory). The methodology + engine were proven on
conceptual domains. This register records where they **strain, break, or need an extension** — the
actual stress-test output. Append-only; each finding gets a disposition.

| # | Finding (issue with the system) | Severity | Disposition |
|---|---|---|---|
| S1 | **Terminal is a *decision*, not concept-mastery.** The methodology's goal/achievement is "understand concept X." Here the goal is "make & justify a setup decision." The concept graph can teach the *mechanisms*, but the payload is a **judgment over tradeoffs** — and tradeoffs are `contrasts`/relations, *not* prerequisites. So the highest-value content lives on the *non-gating* axis the engine treats as secondary. | high | **open** — likely needs a first-class **decision node** (inputs = concepts, output = a justified choice) or a "tradeoff" object. Logged for the engine. |
| S2 | **Surfaces are a parallel menu, not a prerequisite ladder.** CLI / Desktop-Code / VS Code / Cowork / Design / Codex are *alternatives*. The prerequisite relation among them is thin; the concept graph wants depth (a DAG of dependencies), the domain has **breadth** (parallel options, shallow deps). Risk: a forced/artificial prereq chain. | high | **open** — modelled them as siblings under one orientation concept + `contrasts`, not a chain. Tests whether the engine tolerates wide-shallow graphs. |
| S3 | **Caveats/constraints aren't concepts.** "Computer-use is Pro/Max-only," "Desktop reads `claude_desktop_config.json`, the CLI doesn't" — these are *constraints attached to a concept*, not concepts. The data model has no "constraint" object; they land in the definition/confusion, which works but understates load-bearing gotchas. | med | **accepted** — fold into `confusions` + lesson body for now; flag a possible `caveat` field. |
| S4 | **Cross-*tool* (Codex) breaks the single-engine framing.** The framework assumes one system; this domain spans Claude **and** Codex. `portable-skill` + `agents-md` bridge it, but it's a bolt-on, and "one engine, many front-ends" (S2's orientation concept) is *false* once Codex enters. | med | **open** — modelled Codex as a sibling surface + a portability concept; the orientation concept's framing is scoped to *Claude* explicitly. |
| S5 | **Perishability.** This domain moves monthly (Claude Code releases weekly); a Gödel concept graph is static. A research-grounded curriculum needs an `as_of` + refresh policy the engine doesn't model. | med | **open** — note an `as_of` date on the curriculum; ties to `learning_map`'s Ground stage. |
| S6 | **The validator hard-codes the stage count.** `scripts/validate-content.mjs` had `for (let s=0; s<=16; s++)` — godel's 17 stages baked into the *generic* gate. A new instance can't just supply content; it must edit the engine. The "one engine, many curricula" promise leaks. | med | **open** — patched to `0..6` for this instance; the engine should derive the range from `LESSONS`, not hard-code it. Logged as the canonical example of a gate that doesn't travel. |
| S7 | **Cloning the engine drags the first domain's payload.** rsync of godel brought `backend/` (the Gödel LLM judge), `CONTENT_NOTES.md`, `RUNNING_EXAMPLE.md`, the ADRs, and `glossary.ts`/`notation.ts` full of Gödel terms. The framework and its *content fixtures* aren't separable; a real "new instance" must scrub domain artifacts by hand. | med | **open** — left the inert Gödel `glossary.ts`/`notation.ts` in place (my content uses no `@t`/`@n`, so they're harmless data) to avoid scope creep; flags that the engine needs a content/skeleton split (a `create-instance` that ships empty fixtures). |
| S8 | **Framing copy is hard-coded in components, not content.** The sidebar brand, the skill-tree intro banner, the `<title>`, and a `LESSONS.length + 13` progress count (godel's 13 achievements) all lived in `.tsx`, not in a config the curriculum supplies. Each had to be hand-edited; none is derived. | low | **open** — patched all four. The engine should read brand/strings/achievement-count from a per-instance config or from the graph itself. |
| S9 | **The pedagogical *types* are Gödel-shaped.** `types.ts` ships viz kinds `parse-tree`/`parse-explorer`/`coding-encoder`/`godel-loop` and node/edge kinds (`Formula`, `proves`, `satisfies`, `encodes_as`) that are pure Gödel. A decision domain uses only `comparison-table`; the rest are dead weight but still compiled and shape the type surface a new author reads. | low | **accepted** — harmless (unused unions compile); notes that the "typed pedagogy" layer is domain-coupled and a clean engine would factor the universal kinds (table, prose, typed-graph) from domain-specific ones. |

## Slice 2-4 outcome (this run)
The concept-graph methodology **held** on a decision domain: all gates pass (`npm run check` green — 7 stages, 24 concepts, acyclic + closed + group-coherent, 8 graph nodes), the map **derived** correctly (concept→stage backbone via `deriveStageEdges` + transitive reduction), and the rendered skill-tree + concept-graph views are sound. The *content model* fit; the **engine packaging** did not travel cleanly (S6-S8). Net: the methodology is domain-general; the **repo is not yet a reusable engine** — it's godel-with-content-swapped. The highest-value open item remains **S1** (the payload is a judgment over tradeoffs, which the prerequisite DAG models only indirectly — the capstone `cap-setup` carries it, but as an assessment, not as graph structure).
