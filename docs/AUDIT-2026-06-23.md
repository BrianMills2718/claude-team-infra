# Content Audit: 12-Stage Engine-Generated Instance (2026-06-23)

Audit run: 12 per-stage agents against the authoritative compendium. 13 total agents.

## Overall assessment
The generated content has a pervasive accuracy problem: every stage (0-5) contains fabricated specifics — version numbers, environment variable names, file paths, setting keys, and behavioral rules — that do not appear in the authoritative compendium and cannot be verified. Additionally, all six stages have significant target-concept gaps where 2-5 named concepts have zero lesson coverage, and multiple quiz questions mark factually wrong answers as correct, in some cases because the quiz tests content the lesson never taught. The course is not publishable in its current form without a systematic fact-checking pass against authoritative sources and substantial lesson additions.

## Systematic patterns (root causes)
- HALLUCINATED SPECIFICS IN EVERY STAGE: The lesson engine consistently introduces version numbers (v2.1.59+, v2.1.136+, v2.1.142+, v2.1.175+, v2.1.85), environment variable names (CLAUDE_CODE_API_KEY_HELPER_TTL_MS, CLAUDE_CODE_PACKAGE_MANAGER_AUTO_UPDATE, CLAUDE_AGENT_SDK_DISABLE_BUILTIN_AGENTS), file paths (~/.claude/projects/{project}/{sessionId}/subagents/agent-{agentId}.jsonl, /etc/claude-code/managed-settings.json, Windows registry paths), and named setting keys (claudeMd, policyHelper, allowManagedPermissionRulesOnly, minimumVersion, requiredMinimumVersion) that do not appear in the provided authoritative compendium. This occurs in all 6 audited stages and represents the single largest accuracy risk.
- QUIZ CORRECT ANSWERS BUILT ON UNVERIFIED CLAIMS: Across stages 0-4, multiple quiz questions mark answers as correct that are based on claims absent from or contradicted by the compendium. Stage 1 Q4 has the wrong option marked correct (ancestor vs subdirectory confusion). Stage 2 Q1 and Q2 both have wrong answers based on fabricated version-gated behavior and misapplied wildcard restriction. Stage 3 Q2 tests a non-existent http hook type. Stage 4 Q2 tests an unverified evaluation-order claim. The quiz engine is generating plausible-sounding but unverified correct answers.
- TARGET CONCEPTS WITH ZERO LESSON COVERAGE: Every audited stage has 2-5 named target concepts that receive no lesson treatment whatsoever. The missing concepts are then sometimes tested in quizzes (teaching via quiz rather than lesson), creating questions that test untaught content. This suggests the lesson engine does not cross-check lesson output against the target concept list.
- QUIZ EXPLANATIONS REFERENCE CONTENT NOT IN LESSONS: Quiz answer explanations cite specific precedence chains, version numbers, and behavioral rules (e.g., 'Managed deny > user/project deny > Managed ask...', 'local > project > user') that do not appear in any authored lesson section. This creates a systematic gap between what the lesson teaches and what the quiz assumes learners know when they read the explanations.
- FABRICATED BEHAVIORAL RULES PRESENTED AS DOCUMENTED FACT: The lesson engine consistently adds plausible-sounding operational details not in the authoritative source — timestamped settings backups retaining 5 most recent, hot-reload vs restart-required categorizations, hook-vs-rule interaction priority, managed settings deep-merge semantics, topic-file on-demand loading, /compact subdirectory reload behavior. These are presented with the same authority as sourced facts, with no qualification.
- DISTRACTOR QUALITY IS INCONSISTENT: Multiple quiz questions use distractors that are either too easy to dismiss (obviously wrong, not grounded in any real misconception) or internally incoherent (combining two unrelated false claims). The best distractors encode real misconceptions that plausible learners would have; the engine frequently falls back to throwaway wrong answers instead.

## Lesson engine improvements (applied to author-lessons.workflow.js)
- ADD A GROUNDING CONSTRAINT TO THE PROMPT: Every factual claim in a lesson — especially version numbers, environment variable names, file paths, CLI flags, setting key names, and behavioral rules — must be traceable to the provided authoritative source. Add an explicit prompt instruction: 'Before including any specific technical detail (version number, env var name, file path, setting key name, behavioral rule), verify it appears verbatim or paraphrase-equivalent in the provided source material. If you cannot trace it to the source, omit it. Do not add plausible-sounding specifics that are not in the source.'
- ADD A TARGET-CONCEPT COVERAGE GATE TO THE SCHEMA: The lesson output schema should include a required field `targetConceptsCovered: string[]` that lists which target concepts from the stage definition each lesson section addresses. The workflow should validate that every target concept in the stage appears in at least one lesson's `targetConceptsCovered` list before accepting the lesson set. If any target concept is uncovered, the engine must either add a lesson section or explicitly mark the concept as 'deferred to next stage' with a justification.
- REQUIRE SOURCE TRACEABILITY FOR BEHAVIORAL CLAIMS: Add a prompt instruction: 'For every behavioral claim (what happens when X, what the system does if Y, which takes priority when Z), either quote the source directly or cite the section of the provided compendium. Do not infer behavior from general principles or analogies with other systems.' This directly targets the fabricated behavioral rules pattern.
- PROHIBIT FORWARD-KNOWLEDGE IN EXPLANATIONS: Add a prompt instruction: 'Do not introduce technical terms, precedence chains, or mechanisms in lesson body text that are not defined or explained within the same lesson or an earlier lesson in the stage. Every concept used in an explanation must have been explicitly taught first. Scan lesson output for any term or rule cited in an explanation that was not defined in a prior lesson section, and either add the definition or remove the reference.'
- ADD AN 'UNVERIFIED DETAIL' MARKER FIELD TO THE SCHEMA: For each claim the engine is uncertain about, require it to output an `unverifiedClaims: string[]` field listing the specific claims it could not trace to the source. This makes the uncertainty visible to the reviewing human rather than burying it in authoritative-sounding prose.
- REQUIRE A COMMITTED-VS-PERSONAL CONFIG TABLE WHEN COVERING CONFIGURATION: Any lesson stage covering configuration files must include an explicit table mapping each artifact to: committed to source control / gitignored / never commit. This prevents the recurring omission of the never-commit boundary (~/.claude.json) and the committed-vs-personal scope distinction.

## Quiz engine improvements (applied to author-quizzes.workflow.js)
- ADD A LESSON-GROUNDING CHECK FOR EVERY CORRECT ANSWER: Each question's correct answer must cite the lesson section and sentence that teaches the fact being tested. Add a required schema field `correctAnswerSource: { lessonSection: string, quote: string }` for each correct answer. If the engine cannot fill this field from the authored lessons, the correct answer is untestable and must be replaced with a question about content that was taught.
- ADD A COMPENDIUM-GROUNDING CHECK FOR CORRECT ANSWERS: Beyond lesson-grounding, each correct answer must also be traceable to the authoritative compendium. Add a required field `compendiumGrounded: boolean` and a `compendiumQuote: string` for the specific claim the answer rests on. If `compendiumGrounded` is false, the question must be flagged for human review before publication.
- REQUIRE EACH DISTRACTOR TO ENCODE A NAMED MISCONCEPTION: The schema for each distractor option should include a required `misconceptionTargeted: string` field describing the specific plausible-but-wrong belief the distractor encodes. Distractors without a named misconception (e.g., obviously-wrong throwaway options) should be rejected. This directly addresses the inconsistent distractor quality pattern.
- ADD A 'TESTS UNTAUGHT CONTENT' VALIDATION RULE: After quiz generation, run a cross-check: for every concept, term, rule, or claim referenced in any question or explanation, verify it was introduced in an authored lesson section for this stage or a prior stage. Flag any question that references untaught content. This catches the pattern where quiz explanations cite precedence chains or version numbers never mentioned in lessons.
- PROHIBIT SPECIFIC VERSION NUMBERS, ENV VAR NAMES, AND FILE PATHS IN QUESTIONS UNLESS COMPENDIUM-SOURCED: Add an explicit prompt instruction: 'Do not write quiz questions that hinge on specific version numbers (e.g., v2.1.142+), environment variable names, or file paths unless those exact strings appear in the provided authoritative compendium with that exact name. If you want to test version-gated behavior, use behavior-level questions (what does the system do) rather than version-number recall.'
- REQUIRE SCENARIO FRAMING FOR AT LEAST 50% OF QUESTIONS: Add a prompt instruction and schema validation: 'At least half of each stage's quiz questions must present a concrete operational scenario (a team has configured X, a developer is trying to do Y, a CI run produces error Z) that requires the learner to apply knowledge rather than recall declarative facts. Declarative recall questions (which of the following is true?) should be the minority.' This addresses the pattern of multi-select 'select all true statements' questions that test recall instead of application.
- ADD AN ANSWER-CONSISTENCY CHECK AGAINST LESSON EXAMPLES: For any quiz question that uses a code snippet or configuration example, require that the snippet either (a) appeared in the lessons or (b) is independently verified against the compendium. Snippets introduced fresh in a quiz (not previously seen in lessons) have the highest hallucination risk and should require explicit compendium grounding.
