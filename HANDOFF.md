### U1: Wildcard `*` behavior in Bash permission patterns (HIGH PRIORITY)

**What was changed:** Stage-1 preQuiz question "You add `Bash(npm run *)` expecting it to cover `npm run test --watch`. A teammate reports Claude still prompts." — changed `correct` from `1` to `0` and rewrote option A to say `*` DOES match across spaces and the real issue is scope.

**The uncertainty:** It's not definitively established whether `*` in Claude Code Bash permission patterns matches across spaces/multiple arguments (greedy) or only within a single word-component. The previous session concluded `*` is greedy (matches `test --watch` as a sequence including spaces), making the old correct answer (option B: "component-by-component") wrong. BUT the old explanation explicitly said "`*` matches a single path/command component. `Bash(npm run *)` covers `npm run test` but NOT `npm run test --watch`" — which is internally consistent, whether or not it's factually accurate.

**Why it matters:** If `*` is actually component-based (the old explanation), then we just changed a CORRECT answer to WRONG. If `*` is greedy (the new explanation), we correctly fixed a bug.

**How to verify:** Check the Claude Code source or documentation at `code.claude.com/docs/en/settings` under permission rule syntax, or look for the actual pattern-matching implementation. The key question: does `Bash(npm run *)` match `npm run test --watch`?

**Current state of the question (option A):** "`Bash(npm run *)` should cover `npm run test --watch` — `*` matches any character sequence including spaces. The teammate is likely running the command in a context where the allow rule doesn't apply (wrong settings scope, or a managed deny rule overrides it)."

**If `*` is actually component-based (old behavior was correct):** Revert to `correct: 1` and restore the original option A text ("`*` in Bash patterns matches across multiple arguments..."), or better, rewrite the question entirely to avoid the ambiguity.

## The Research Document analysis (2026-06-24)

Brian asked me to review `/home/brian/projects/Claude Teams Configuration Research.md` for inconsistencies with actual documentation. The document is an enterprise deployment guide for Claude Desktop + Cowork + Teams admin. 

**Errors found (in decreasing confidence):**

1. **Plugin directory: `.claude-plugin/plugin.json` is wrong.** `plugin.json` lives at the ROOT of the plugin directory, not inside a `.claude-plugin/` subdirectory. The document adds an extra level of indirection that doesn't exist.

2. **`disable-model-invocation: true` YAML key doesn't exist.** The concept (prevent the model from auto-triggering a skill) is real, but this key name does not appear in the official Claude Code skills frontmatter spec. `user-invocable: true/false` IS real. Someone may have invented or hallucinated this key.

3. **`monitors/monitors.json` and `.lsp.json` in plugin spec.** These don't appear in the official plugin manifest documentation I know. They may be extrapolated or fabricated.

4. **Level 3 skill reference files are NOT auto-loaded.** The document describes a three-level runtime that auto-loads referenced files when a process step calls for them. That's a design pattern recommendation, not actual runtime behavior. Files must be explicitly read during execution.

5. **`~/.claude/agents/` is not a standard directory.** The diagram implies `agents/` is a standard `~/.claude/` subdirectory for Claude Code. It's a plugin-internal directory concept, not a global config directory.

6. **Team vs. Enterprise plan boundary** is stated inconsistently within the document itself.

**What to do with this:**
- The document appears to be source material for the course. The course should NOT teach the wrong plugin structure or the non-existent `disable-model-invocation` key.
